# HANDOVER — working notes for this repo

Practical techniques for an agent continuing work here. Focus is *how to work
with the files*, not feature wishlists.

## 1. What's in the repo

- `*.gpx` — pilgrim trail tracks (Gudbrandsdalsleden). Generated by an external
  tool (`creator="pilegrimsleden-extractor"`). `oslo_eidsvoll.gpx` is the active
  one; it now also contains `<wpt>` POI waypoints.
- `oslo_eidsvoll.html` — a **single self-contained** Leaflet map viewer for the
  Oslo→Eidsvoll track. All code, data, and styles are inline. This is where most
  work happens.
- `oslo_eidsvoll.zip` — the gpx zipped (Outdooractive imports `.zip` of `.gpx`).
- Git repo with short imperative commit messages (e.g. "Add POI to Oslo Eidsvoll
  map"). Match that style.

## 2. The big-data-line gotcha (read before editing the HTML)

`oslo_eidsvoll.html` has two **enormous single lines** near the top of the
`<script>`:

- `var _track = [[lat,lon,ele],...]` — ~46 KB on one line (~1538 points).
- `var _pois  = [{...},...]` — the POI array, ~23 KB on one line.

Consequences and how to cope:

- **Do not `view` the whole file** — it exceeds the 20 KB read cap and truncates.
  Use `view` with `view_range`, or `grep`/`awk` to locate things.
- **Find anchors by line number**, e.g.:
  ```bash
  awk '/^var _track = \[\[/{print NR": data"} /^<\/script>/{print NR": end"} \
       /^function initMap/{print NR} /^try \{/{print NR}' oslo_eidsvoll.html
  ```
- **Line numbers drift after every edit.** Re-run the anchor `awk`/`grep` before
  any slice or range operation. Never reuse stale line numbers.
- To **insert near the data lines**, prefer a tiny Python script (read → split on
  `\n` → `list.insert` → write) over the `edit` tool, because the data line is too
  large to reproduce in an `old_str`.
- To **extract track/POI data** for analysis, regex the raw text (preserves exact
  coordinate strings/precision) rather than reformatting through a parser.

## 3. Headless testing harness (no browser available)

There is no browser and Node isn't preinstalled. The environment is `aarch64`,
and `apt`/`pip`(default) are restricted, but **outbound HTTPS works**.

- A standalone Node is already downloaded at:
  `/tmp/node-v20.18.0-linux-arm64/bin/node`
  (if missing: `curl -sL https://nodejs.org/dist/v20.18.0/node-v20.18.0-linux-arm64.tar.xz | tar xJ -C /tmp`)
- Use it for `node --check` (syntax) and to **run the page's JS logic with mocked
  globals**. The script is plain ES5-ish browser JS; you can execute the body by
  providing fake `L` (Leaflet), `document`, and `window`.

Pattern that works well:

1. Find the script range with `awk` (data line .. `</script>`, or .. end of the
   function you want to test).
2. `sed -n 'START,ENDp' oslo_eidsvoll.html` into a temp file, prepended with mocks
   and appended with assertions.
3. Mock `L.map` so `map.on("click", fn)` **captures the handler**; then call
   `fn({ latlng: { lat, lng } })` to drive interactions. Mock
   `L.marker/circleMarker/polyline/rectangle/control/...` to record what got
   created. Mock `document.getElementById` with an `innerHTML` setter to capture
   panel output.

Two slicing modes:
- **Pure functions** (`computeStats`, `tilesAlongTrack`, tile math): slice up to
  *just before* `function initMap` and call them directly — don't execute `initMap`.
- **Interaction logic** (click handlers): include through end of `initMap` **and**
  the `try { initMap(); }` call so handlers register; provide full mocks. If you
  stop mid-`try`, you'll get "Unexpected end of input" from unbalanced braces — cut
  at the end of `initMap` (line `}` before `try {`) and add your own
  `} catch(e){}` if you included `try {`.

Verify numeric logic against an independent **Python** reimplementation (Python is
available). This caught/confirmed the tile-count math and POI distances.

## 5. GPX handling

- **Namespace trap:** the upstream extractor emits *broken* GPX — `<trk>`,
  `<metadata>` are in **no namespace** while `<trkpt>`/`<wpt>` use an `ns0:` prefix
  bound to the GPX URI. Strict parsers (Garmin/Outdooractive) then ignore the
  track. The fix is a full rewrite to clean GPX 1.1: declare a **default**
  `xmlns="http://www.topografix.com/GPX/1/1"`, add `xsi:schemaLocation`, put every
  element in that namespace.
- **Element order matters for XSD.** Waypoint children must be in schema order:
  `name, desc, link, sym, type` (and `ele` first for trackpoints). Wrong order =
  invalid.
- **Validate against the official schema** (don't eyeball it):
  ```bash
  pip install lxml --break-system-packages         # default pip is blocked without this
  curl -sL https://www.topografix.com/GPX/1/1/gpx.xsd -o /tmp/gpx.xsd
  python3 -c 'import lxml.etree as E; \
    print(E.XMLSchema(E.parse("/tmp/gpx.xsd")).validate(E.parse("oslo_eidsvoll.gpx")))'
  ```
- When rewriting, **preserve original trackpoint coordinate strings** (regex them
  out verbatim) so precision/elevation don't drift. Cross-check counts
  (trkpt/wpt) and that data is byte-identical before/after.
- `<sym>` uses Garmin's named symbols (e.g. `Church`, `Lodging`, `Restaurant`,
  `Shopping Center`, `Picnic Area`, `Campground`, `Information`, `Museum`, `Park`).
  Omit `sym` for unmapped categories rather than inventing names.

## 6. POI data source (pilegrimsleden.no)

If POIs need refreshing, the tour planner is a Craft CMS app whose JSON API was
reverse-engineered from its bundle:

- Download `https://www.pilegrimsleden.no/resources/js/tourplanner.min.js` and
  grep for `actions/` to find endpoints.
- `…/actions/tourplanner/trails/get-trails` → list of trails. The URL param
  `activeTrail=449482` is a **Craft entry id** ("Gudbrandsdalsleden øst"); its
  `trail.id` is **212** — that inner id is what the POI endpoint needs.
- `…/actions/pilegrimsleden/poi/list?trailId=212` → ~822 POIs for the whole
  Oslo→Trondheim route. Fields: `t` title, `lt`/`ln` lat/lon, `d` desc, `c`
  category, `u` url, `i` image.
- This covers the entire route; **filter to the local segment by distance to the
  track** (haversine to nearest trackpoint). A natural cutoff was visible in the
  sorted distances (~3 km here). Sort/place waypoints by along-track position.
- Send `-H 'X-Requested-With: XMLHttpRequest'`. OSM/this API allow CORS.

## 7. HTML viewer architecture (function map)

Order in the `<script>` (re-grep for current line numbers):
`toRad/haversine/formatDist` (geo) · `hexToRgb/elevColor` (unused gradient kept for
later) · `computeStats` · `cell/renderClickInfo` (top-right panel) ·
`escapeHtml/clearPosition/showPoiInfo/addPois` (red POI dots, click to pin) ·
`addLocateControl` (geolocation → thin red ring) · `makeDotIcon/drawTrack/
findNearest` · offline cache block: `OFFLINE_*` consts, `openTileDB/idbGet/idbPut`,
`dbPromise`, `lon2tileX/lat2tileY`, `tilesAlongTrack`, `CachedTileLayer`,
`downloadTiles`, `offlineStatus`, `addOfflineControl` · `initMap` ·
`try { initMap() }`.

Conventions:
- `trackPoints` are `[lat, lon, ele]`; POIs are `{lat, lon, name, desc, type, link}`.
- The map click handler captures everything; it computes nearest trackpoint and
  uses a **pixel** threshold (`latLngToContainerPoint`, ~20 px) to decide
  "on path" vs "free" click. Distances: "Dist" = straight line between clicks,
  "Path" = along-track between nearest-path points.
- `CachedTileLayer` overrides `createTile` to serve from IndexedDB then network
  (caching the blob). `tilesAlongTrack(minZ, maxZ)` returns one tile per GPX point
  (the containing tile), deduped across all zooms — a 1-tile-wide ribbon caching
  the whole trail (~3,300 tiles / ~64 MB for z10-19). The download button caches
  this once; throttled via `downloadTiles`.

## 8. Environment / runtime caveats

- **Shared environment:** files can change under you (a `.gpx` was gzipped to
  `.gpx.gz` by something external mid-session). Re-check state; don't assume.
- **Secure-context features:** geolocation and IndexedDB need `https://` or
  `localhost`. Serve with `python3 -m http.server` for real testing; `file://`
  may disable them (the code degrades gracefully).
- **OSM tiles:** `tile.openstreetmap.org` sends `Access-Control-Allow-Origin: *`
  (so `fetch`+cache works) but its usage policy **forbids bulk downloading**. The
  whole-trail cache is ~3,300 requests (a one-time prep) and is throttled (4
  workers, 25 ms spacing) to be gentle; the on-view layer otherwise caches only
  what you actually look at. The offline model evolved during development
  (1 km buffer → per-view pyramid → current 1-tile-per-point ribbon); git history
  has the alternatives if a different trade-off is ever wanted.
- Minor quirk: `"use strict"` sits *after* the `var _track` line, so strict mode
  isn't actually in effect. Harmless; just don't rely on it.

## 9. Suggested loop when changing the HTML

1. `awk` for anchors → know current line numbers.
2. Make the edit (`edit` tool for code; Python insert for anything touching the
   data lines).
3. `node --check` the file (slice out the script if needed).
4. Run a mocked-`L` harness to exercise the changed handler/function.
5. For numeric changes, cross-check with a quick Python computation.
6. Re-grep to confirm no stale references to renamed/removed symbols.
